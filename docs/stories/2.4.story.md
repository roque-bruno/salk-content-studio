# Story 2.4 — Migrar persistencia para SQLite

**Epico:** 2 — Fundacao e Deploy
**Status:** Done
**Executor:** @dev (Dex)
**PRD:** docs/prd/PRD-content-studio-v3.md

## Descricao

Substituir os JSON files usados para persistencia (pieces.json, reviews.json, metrics.json, calendars/) por um banco SQLite. Isso resolve problemas de concorrencia, permite queries, e e compativel com Render (arquivo persistente no disco).

## Acceptance Criteria

- [ ] Banco SQLite criado em `data/studio.db`
- [ ] Tabelas criadas: `pieces`, `reviews`, `metrics`, `calendars`, `calendar_slots`
- [ ] Modulo `database.py` criado com funcoes CRUD usando `aiosqlite`
- [ ] `services.py` atualizado para usar `database.py` em vez de JSON files
- [ ] Migration automatica: se JSON files existem, importar dados para SQLite na primeira execucao
- [ ] Todos os endpoints da API continuam funcionando identicamente
- [ ] Testes unitarios para operacoes CRUD do database
- [ ] Performance: queries < 50ms para operacoes comuns

## Tasks

1. Instalar `aiosqlite` como dependencia
2. Criar `database.py` com schema e funcoes CRUD
3. Implementar migration JSON → SQLite
4. Atualizar `services.py` para usar database
5. Testar todos os endpoints
6. Escrever testes unitarios
7. Remover logica de JSON files (manter como fallback temporario)

## Notas Tecnicas

- SQLite e ideal: zero config, arquivo unico, suporta concorrencia limitada (suficiente para 1 usuario)
- `aiosqlite` para compatibilidade com FastAPI async
- Schema deve incluir timestamps (created_at, updated_at) em todas tabelas
- Indices em: pieces.stage, pieces.brand, reviews.verdict, metrics.platform
